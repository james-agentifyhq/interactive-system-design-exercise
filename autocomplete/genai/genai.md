# Autocomplete — GenAI System Design

Traditional autocomplete matches prefixes against a static dataset. GenAI autocomplete *generates* completions by understanding context — it doesn't look up what you're typing, it predicts what you're *about to* type. This is a fundamentally different problem with different architecture, constraints, and tradeoffs.

## Question
Design an AI-powered autocomplete system that generates contextual suggestions as the user types. Unlike traditional autocomplete that matches against a fixed dataset, this system uses a language model to produce completions that understand the user's intent.

Real-world examples:
- **GitHub Copilot**: Code completion based on current file, open tabs, language, and comments
- **Gmail Smart Compose**: Sentence-level completion based on email context and recipient
- **Google Search AI suggestions**: Query refinements generated by understanding intent, not just prefix matching
- **Notion AI / Google Docs**: Continue writing based on document context
- **ChatGPT/Claude suggested follow-ups**: Conversation-aware next-question suggestions

## Requirements

### Functional Requirements
- Generate contextual completions as the user types
- Support streaming — show partial results as the model generates tokens
- Completions should be relevant to the current context (document, conversation, code file)
- User can accept (Tab), dismiss (Escape), or ignore suggestions
- Support multiple suggestion candidates (e.g., Copilot shows 3 alternatives)

### Non-Functional Requirements
- **Low perceived latency**: First token should appear within ~200ms (users tolerate more than traditional autocomplete because the quality is higher)
- **Cost efficiency**: LLM inference is expensive — can't call the model on every keystroke
- **Graceful degradation**: If the model is slow or unavailable, fall back to traditional autocomplete or show nothing
- **Privacy**: User context sent to the model may contain sensitive data

### Requirements Exploration

#### How much context does the model need?
More context = better suggestions, but also more tokens = higher cost and latency.
- Gmail: Current email + recipient info (~500 tokens)
- Copilot: Current file + relevant open files (~2000-8000 tokens)
- Document editor: Surrounding paragraphs + document title (~1000 tokens)

#### Should completions be single-word, sentence, or multi-line?
Depends on the product:
- Search box: Single query completion (one line)
- Email: Rest of the current sentence (Smart Compose style)
- Code editor: Multiple lines, potentially entire function bodies

#### Is the model self-hosted or API-based?
- **API-based** (OpenAI, Anthropic, Google): Simpler ops, pay-per-token, higher latency due to network hop
- **Self-hosted** (vLLM, TGI, Ollama): Full control over latency, fixed cost, requires GPU infrastructure
- **On-device** (Apple Intelligence, Copilot local mode): Zero network latency, limited model size, privacy-preserving

## Architecture

```
                    ┌─────────────┐
                    │   Client    │
                    │  (Editor)   │
                    └──────┬──────┘
                           │ Context + partial input
                           │ (WebSocket or SSE stream)
                           ▼
                    ┌─────────────┐
                    │  API Gateway│  ← Auth, rate limiting, routing
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  Completion │  ← The orchestrator
                    │  Service    │
                    └──┬────┬─────┘
                       │    │
            ┌──────────▼┐  ┌▼───────────┐
            │ Semantic   │  │  Prompt     │
            │ Cache      │  │  Builder    │
            │            │  │             │
            └────────────┘  └──┬─────┬───┘
                               │     │
                               │  ┌──▼───────────┐
                               │  │ Retrieval     │  ← Optional (RAG)
                               │  │ Layer         │     Vector search for
                               │  │ (Embedding    │     relevant context
                               │  │  Store)       │
                               │  └──┬────────────┘
                               │     │ retrieved context
                               ├─────┘
                               │ prompt
                        ┌──────▼──────┐
                        │  LLM        │  ← Model inference
                        │  (streaming)│     Token-by-token output
                        └─────────────┘
```

### Component Responsibilities

- **Client (Editor)**: Captures user input and context, sends to completion service, renders streaming suggestions inline. Handles accept/dismiss interactions.
- **[API Gateway](../../concepts/backend/api-gateway.md)**: Authentication, per-user [rate limiting](../../concepts/rate-limiting.md), request routing. May also handle billing/usage tracking.
- **Completion Service**: The "brain" — decides WHEN to trigger a completion (not every keystroke), builds the prompt from context, checks the semantic cache, calls the LLM if needed, and streams results back.
- **Prompt Builder**: Constructs the prompt from raw context. Selects relevant context, truncates to token budget, formats for the model. This is where prompt engineering lives.
- **[Retrieval Layer](../../concepts/genai/rag.md) (optional)**: Searches an [embedding store](../../concepts/genai/embeddings-and-vector-search.md) for context relevant to the user's input. Feeds retrieved documents/snippets to the Prompt Builder so the LLM can generate completions grounded in real data. Only needed when the LLM's training data isn't sufficient (enterprise data, private codebases, domain-specific knowledge).
- **Semantic Cache**: Unlike traditional caching (exact key match), this caches by *meaning*. Similar contexts should return cached results. Uses embedding similarity.
- **LLM**: The model that generates completions. Could be a hosted API or self-hosted inference server. Streams tokens via SSE or WebSocket.

### Architecture Patterns

The architecture above supports three patterns depending on whether the Retrieval Layer is present:

**Pattern 1: Pure Generation** (no retrieval)
The LLM generates completions purely from its training knowledge + the immediate context (text before/after cursor). No embedding store needed. The model's weights *are* the knowledge store.
- Example: Gmail Smart Compose, basic code completion
- Pros: Simple, low latency (no retrieval step)
- Cons: Can't access private/recent data

**Pattern 2: RAG-Augmented Completion** (retrieval required)
User input is embedded → vector search retrieves relevant documents → retrieved context is injected into the prompt → LLM generates completions grounded in that data.
- Example: Enterprise search autocomplete, Slack-aware suggestions
- Pros: Grounded in real data, no hallucination of company-specific terms
- Cons: Added latency (~50-100ms for retrieval), requires maintaining an embedding index

**Pattern 3: Hybrid** (retrieval for context, generation for completion)
The retrieval layer fetches *context* (relevant files, docs), but the LLM *generates* the actual completion. This is how modern code assistants work — they index your codebase, retrieve related files, and use them as context for generation.
- Example: Cursor (indexes your repo), Copilot Workspace, Cody
- Pros: Best quality — grounded + creative
- Cons: Most complex, highest latency, requires codebase indexing pipeline

```
Pattern 1:  Context → Prompt Builder → LLM → Completion
Pattern 2:  Query → Retrieval → Prompt Builder → LLM → Completion
Pattern 3:  Context → Retrieval (related files) → Prompt Builder → LLM → Completion
```

## Data Model

### Completion Request
```typescript
interface CompletionRequest {
  context: {
    textBefore: string;      // Text before the cursor
    textAfter: string;       // Text after the cursor (if any)
    documentTitle?: string;  // Document/file name for context
    language?: string;       // Programming language or natural language
    metadata?: Record<string, string>;  // App-specific context
  };
  options: {
    maxTokens: number;       // Token budget for completion (e.g., 50-200)
    temperature: number;     // 0 = deterministic, 0.7 = creative
    n: number;               // Number of alternatives to generate
    stopSequences: string[]; // Stop generation at newline, period, etc.
  };
}
```

### Completion Response (streamed)
```typescript
interface CompletionChunk {
  id: string;               // Request ID for cancellation
  text: string;             // New token(s) in this chunk
  done: boolean;            // Is generation complete?
  finishReason?: 'stop' | 'length' | 'cancelled';
  usage?: {
    promptTokens: number;
    completionTokens: number;
    latencyMs: number;
  };
}
```

### Embedding Store Entry (Patterns 2 & 3)
```typescript
interface EmbeddingDocument {
  id: string;                // Document/chunk identifier
  content: string;           // Original text content
  embedding: number[];       // Vector embedding (768-1536 dimensions)
  source: string;            // Where this came from (file path, URL, etc.)
  updatedAt: number;         // For incremental re-indexing
  metadata?: {
    language?: string;       // Programming language, natural language
    type?: string;           // "code", "docs", "comment", "config"
  };
}
```

### Cache Entry
```typescript
interface SemanticCacheEntry {
  embedding: number[];       // Vector embedding of the context
  completion: string;        // The generated completion
  model: string;             // Which model generated this
  createdAt: number;
  hitCount: number;
}
```

## Deep Dive: When to Trigger Completion

This is the most critical design decision — and very different from traditional autocomplete.

### The Problem
Traditional autocomplete fires on every keystroke (with debouncing). GenAI can't do this because:
1. **Cost**: Each LLM call costs tokens (~$0.001-0.01 per completion). At 5 keystrokes/second, that's $0.3-3/minute per user.
2. **Latency**: Even fast models take 200-500ms for first token. Calling on every keystroke creates overlapping requests.
3. **Wasted generation**: If the user is mid-word, the completion will be wrong by the time it arrives.

### Trigger Strategies

**1. Pause-based (most common)**
Trigger after the user pauses typing for N ms. Similar to [debouncing](../../concepts/frontend/debouncing-and-throttling.md) but with longer delays (500-1000ms vs 300ms for traditional).
- GitHub Copilot uses ~500ms pause
- Gmail Smart Compose uses ~300-500ms
- Pros: Simple, natural pause = user is thinking = good time to suggest
- Cons: Feels slow for fast typers

**2. Semantic triggers**
Trigger at meaningful boundaries:
- After typing a period, colon, opening brace, newline
- After typing a comment (`//`, `#`, `"""`)
- After typing a function signature
- Pros: Higher quality suggestions at the right moment
- Cons: More complex trigger logic, language-specific

**3. Speculative pre-fetching**
Start generating *before* the user pauses:
- On every pause > 200ms, speculatively generate
- Cache the result locally
- If the user keeps typing and the result still matches, show it
- If not, discard silently
- Pros: Feels instant to the user
- Cons: Expensive — many generations are wasted

**4. Hybrid**
Combine pause-based + semantic triggers. Use cheap heuristics to decide if a completion is likely useful, then trigger the expensive LLM call only when confidence is high.

## Deep Dive: [Streaming](../../concepts/genai/streaming-inference.md)

### Why Streaming Matters
Traditional autocomplete returns results all at once. GenAI must stream because:
- A 50-token completion at 30 tokens/second takes 1.7s to fully generate
- Without streaming, the user waits 1.7s and sees nothing
- With streaming, the first token appears in ~200ms and the rest flows in

### Implementation: Server-Sent Events (SSE)
```
Client                          Server
  │                               │
  │── POST /completions ─────────▶│
  │                               │── prompt → LLM
  │◀── data: {"text": "const"} ──│◀── token: "const"
  │◀── data: {"text": " sum"} ───│◀── token: " sum"
  │◀── data: {"text": " ="} ─────│◀── token: " ="
  │◀── data: {"done": true} ─────│
  │                               │
```

### Cancellation
The user keeps typing while the model is generating → the completion is now stale. We must cancel:
- **Client-side**: Close the SSE connection / abort the fetch ([AbortController](../../concepts/frontend/race-conditions.md))
- **Server-side**: Stop model inference to free GPU resources (not just ignore the output)
- This is different from traditional autocomplete where we argued against aborting — here, inference is expensive and we actively want to stop it

### Alternatives to SSE
- **WebSocket**: Bidirectional, useful if the server also needs to push context updates. More complex connection lifecycle.
- **Raw HTTP streaming**: Use `fetch()` with a `ReadableStream` and parse chunks yourself — no `EventSource` API, no auto-reconnect, no `Last-Event-ID`. Lighter protocol overhead but you reimplement those features yourself. Works over HTTP/2 and HTTP/3 equally.
- **gRPC streaming**: Best for internal service-to-service. Used by Copilot internally. Strong typing via protobuf.
- **Realtime database (Firebase RTDB, Firestore, Supabase Realtime)**: Server writes tokens to a DB path, client subscribes. Decouples producer from consumer and adds persistence, but adds latency (extra hop) and cost (per-write charges at token-by-token granularity). Better suited for chat/collaboration than high-frequency token streaming.

**Note on HTTP/3**: SSE, WebSocket, and raw streaming all benefit from HTTP/3 (QUIC) transparently — no application code changes. Key improvements: no head-of-line blocking across multiplexed streams, connection migration across network changes (wifi → cellular), and 0-RTT reconnection. These reduce SSE reconnection frequency but don't eliminate the need for `Last-Event-ID` / retry logic.

## Deep Dive: [Semantic Caching](../../concepts/genai/semantic-caching.md)

### The Problem with [Traditional Caching](../../concepts/caching.md)
Traditional cache: `"san" → [San Francisco, San Jose, ...]` — exact key match.

GenAI cache can't use exact match because:
- Context is long and varies slightly every time (cursor position, one more word typed)
- Two nearly identical contexts should return the same completion
- The "key" is the *meaning* of the context, not its exact text

### How Semantic Caching Works
1. When a completion is requested, compute an **embedding** of the context
2. Search the cache for entries with **similar embeddings** (cosine similarity > threshold)
3. If a similar entry exists → return cached completion (cache hit)
4. If not → call the LLM → store the result with its embedding

```
New request context:  "function calculateTotal(items) {"
                              ↓ embed
                      [0.23, -0.81, 0.44, ...]
                              ↓ similarity search
Cache:
  [0.24, -0.80, 0.45, ...] → "return items.reduce((sum, i) => sum + i.price, 0);"  ← 0.97 similarity ✓
  [0.91, 0.12, -0.33, ...] → "console.log('hello');"                                ← 0.21 similarity ✗
```

### Tradeoffs
- **Similarity threshold**: Too high (0.99) = almost no hits. Too low (0.8) = returns wrong completions.
- **Embedding model cost**: Computing embeddings is cheaper than full LLM inference but still costs tokens.
- **Cache storage**: Embeddings are vectors (768-1536 dimensions). Need a vector database (Pinecone, pgvector, Qdrant) or in-memory index (HNSW).
- **Freshness**: Code context changes rapidly. Cache entries go stale quickly. Short TTL (minutes, not hours).

### When Semantic Caching Helps Most
- **Boilerplate code**: `import React from 'react'` — same completion every time
- **Common patterns**: Error handling, logging, CRUD operations
- **Repeated contexts**: User undoes and retypes — same completion should appear instantly

### When It Doesn't Help
- **Highly specific code**: Unique variable names, custom logic
- **Creative writing**: Each sentence is unique
- **Conversation**: Context changes with every message

## Deep Dive: [Prompt Engineering](../../concepts/genai/prompt-engineering.md) for Autocomplete

### Prompt Structure
```
[System instruction]
You are a code completion engine. Complete the code at the cursor position.
Only output the completion, nothing else. Do not repeat existing code.

[Context — before cursor]
function calculateTotal(items: CartItem[]): number {
  const subtotal = items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  const tax = |

[Context — after cursor]
  return subtotal + tax + shipping;
}
```

### Key Prompt Design Decisions

**1. Fill-in-the-middle (FIM) vs left-to-right**
- Left-to-right: Model only sees text before cursor. Simpler but ignores what comes after.
- FIM: Model sees text before AND after cursor. Much better completions. Requires model support (most modern models support `<|fim_prefix|>`, `<|fim_suffix|>`, `<|fim_middle|>` tokens).

**2. Context window budget**
Models have a finite context window (4K-128K tokens). How to allocate:
- System prompt: ~100 tokens (fixed)
- Text before cursor: 60-70% of remaining budget
- Text after cursor: 20-30%
- Related files / documentation: 10% (if room)

**3. Stop sequences**
Control when the model stops generating:
- Single-line completion: Stop at `\n`
- Multi-line completion: Stop at `\n\n` (blank line) or end of block
- Sentence completion: Stop at `.` or `?`

**4. Temperature**
- Code: 0 to 0.2 (deterministic — there's usually one right answer)
- Text/email: 0.3 to 0.7 (some creativity is desirable)
- Creative writing: 0.7 to 1.0

## Deep Dive: [Cost Optimization](../../concepts/genai/llm-cost-optimization.md)

### The Cost Problem
At scale, GenAI autocomplete is orders of magnitude more expensive than traditional:

| | Traditional | GenAI |
|---|---|---|
| Per-query cost | ~$0.000001 (CPU) | ~$0.001-0.01 (GPU/API) |
| 1M queries/day | ~$1 | ~$1,000-10,000 |

### Cost Reduction Strategies

**1. Trigger filtering (biggest impact)**
Don't call the model when a completion is unlikely to be useful:
- User is mid-word → skip (wait for word boundary)
- User is deleting text → skip
- Cursor is in a string literal or comment → maybe skip
- Document is very short (no context) → skip

Good filtering can reduce LLM calls by 60-80%.

**2. [Model tiering](../../concepts/genai/model-tiering.md)**
Use a small, fast, cheap model for most completions. Escalate to a large model only when:
- The small model's confidence is low
- The user explicitly requests suggestions
- The context is complex (long function, unusual pattern)

```
User types → Small model (Haiku-class, ~$0.0003/completion)
                  ↓ low confidence
             Large model (Sonnet-class, ~$0.003/completion)
```

**3. Caching (semantic + exact)**
- Exact cache for identical contexts (free)
- Semantic cache for similar contexts (embedding cost only)
- Target: 20-40% cache hit rate

**4. Token budget management**
- Shorter prompts = cheaper. Aggressively truncate context to what's relevant.
- Shorter completions = cheaper. Default to single-line unless the user requests more.
- Use `max_tokens` to cap generation length.

**5. Batching**
If the server handles many users, batch multiple completion requests into a single model call (supported by vLLM, TGI). Reduces per-request overhead.

## Deep Dive: Latency Optimization

### Latency Breakdown
```
Total latency = Network (client→server) + Queue wait + Prompt processing + Token generation
                     ~20ms                  ~0-500ms      ~50-200ms         ~30ms/token
```

### Optimization Techniques

**1. [Speculative decoding](../../concepts/genai/speculative-decoding.md)**
Use a small draft model to generate candidate tokens quickly, then verify them with the large model in a single forward pass. Can generate 2-4x faster.

**2. KV cache reuse**
If the context is similar to a previous request (user added one character), reuse the model's key-value cache from the previous inference. Skips re-processing the entire prompt.

**3. Quantization**
Run the model at lower precision (INT8, INT4) for faster inference with minimal quality loss. Common for self-hosted deployments.

**4. Edge deployment**
Run a small model on-device (phone, laptop) for instant suggestions. Use the cloud model only for complex completions.

**5. Priority queuing**
Active user typing → high priority (needs response in 200ms). Background pre-fetch → low priority. Prevents pre-fetches from blocking interactive requests.

## Deep Dive: [Evaluation and Quality](../../concepts/genai/evaluation-metrics.md)

### How to Measure Quality

**1. Acceptance rate**
What percentage of shown suggestions does the user accept (Tab/Enter)?
- Good: > 30% acceptance rate
- GitHub Copilot reports ~30% acceptance rate

**2. Completion persistence**
Of accepted completions, how many does the user keep vs immediately edit/delete?
- If users frequently edit accepted completions, quality is poor

**3. Characters saved**
Total characters in accepted completions ÷ total characters typed. Measures productivity gain.

**4. Time to first token**
P50 and P95 latency for the first streamed token. Target: < 300ms P50.

**5. User satisfaction**
Thumbs up/down on suggestions (explicit signal). Session-level: does the user keep the feature enabled?

### A/B Testing
- Compare models (GPT-4o-mini vs Claude Haiku vs fine-tuned Llama)
- Compare trigger strategies (pause-based vs semantic)
- Compare context window sizes (1K vs 4K vs 8K tokens)
- Metric: acceptance rate × completion persistence
